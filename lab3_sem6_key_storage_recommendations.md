## LAB3_SEM6 — рекомендации по хранению ключей шифрования

### Что нельзя делать
- **Не хранить ключи/пароли в таблицах БД** “как есть” (даже с RLS): при компрометации БД утекут и данные, и ключи.
- **Не коммитить ключи** (private key, symmetric key) в Git.
- **Не передавать ключи в явном виде** в логируемых местах (SQL-логи, audit-таблицы, CI-логи).

### Как делать безопаснее (практика)
- **Docker secrets / Kubernetes secrets**: ключи лежат в файлах в контейнере (`/run/secrets/...`), права чтения минимальные.
- **Переменные окружения + секрет‑менеджер**: ключи выдаются приложению из Vault/KMS, в БД не попадают.
- **KMS (AWS KMS / GCP KMS / Azure Key Vault)**: БД/приложение получает ключ “по запросу”, возможна ротация и аудит.
- **Ротация ключей**: плановая смена ключей и пере-шифрование данных по расписанию.

### Компромисс для учебного проекта (как у нас в LAB3_SEM6)
- Ключи лежат **на стороне сервера** в файловой системе контейнера (`/etc/postgresql/keys/...`) с правами `600`.
- Это удобно для демонстрации, но в реальных системах лучше использовать secrets/KMS.
