## LAB3_SEM6 — Задание 6: методы шифрования, плюсы/минусы и влияние на производительность

### 1) Хеширование паролей (password hashing)
**Что это:** пароли пользователей БД не шифруются “обратно”, а хранятся как хеш. При входе сравнивается хеш введённого пароля с сохранённым.

**Методы, которые встречаются в PostgreSQL:**
- **MD5**: старый метод, быстрее, но криптографически слабее.
- **SCRAM-SHA-256**: современный метод, устойчивее, лучше защищает от перебора и утечек.

**Плюсы для предметной области (почтовые отделения):**
- защищает аккаунты сотрудников и аудитора (актуально, т.к. доступ к персональным данным есть у ограниченных ролей).

**Минусы:**
- при слабых паролях никакой метод не спасёт от перебора; нужно сочетать с политиками паролей и ограничением попыток.

**Вывод:** для проекта выбран **SCRAM-SHA-256** (современный и рекомендуемый).

### 2) Шифрование выбранных столбцов (column-level encryption)
**Что это:** отдельные чувствительные поля (например телефон, паспортные данные, адрес отделения) хранятся в таблице в виде шифртекста.

**Использованные подходы:**
- **Симметричное шифрование (pgp_sym_encrypt / pgp_sym_decrypt)**  
  Один ключ шифрует и расшифровывает. Удобно для массовых полей (телефон/адрес).
- **PGP с открытым ключом (pgp_pub_encrypt / pgp_pub_decrypt)**  
  Шифрование публичным ключом, расшифровка приватным. Удобно, когда расшифровка должна быть доступна только очень узкому кругу.

**Плюсы для предметной области:**
- снижает риск утечки персональных данных (телефон/паспорт) из дампа/бэкапа/скринов таблиц.
- даже если кто-то увидит таблицу, без ключа он увидит шифртекст.

**Минусы:**
- запросы по зашифрованным полям (поиск/фильтрация) становятся сложнее и часто медленнее: нужно расшифровывать данные “на лету”.
- индексы по шифртексту обычно не помогают для поиска по исходному значению (если нет специальной схемы вроде детерминированного шифрования/хеширования).
- появляется задача безопасного хранения/ротации ключей.

---

### 3) Шифрование при передаче по сети (SSL/TLS)
**Что это:** соединение клиент↔сервер защищено TLS, данные в пути (пароли, результаты запросов) недоступны для перехвата в сети.

**Плюсы для предметной области:**
- критично для удалённых подключений сотрудников/администратора/pgAdmin.
- защищает логины/пароли и результаты запросов, где могут быть персональные данные.

**Минусы:**
- небольшие накладные расходы на шифрование трафика, особенно при больших объёмах передачи данных.
- усложняет администрирование (сертификаты, срок действия, доверие).

### 5) Влияние на производительность (по результатам тестов)
**Шифрование столбцов:**
- запросы, где надо расшифровывать поле в `WHERE`, обычно работают медленнее, т.к. появляются вычисления `...decrypt(...)` и чаще используется последовательное чтение.

**SSL/TLS:**
- на простых запросах разница обычно небольшая,
- на больших выгрузках (много строк/крупный payload) накладные расходы заметнее, т.к. шифруется весь сетевой трафик.
