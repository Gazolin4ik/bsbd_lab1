## Лабораторная работа №2

### Цель
Реализовать безопасные операции с точечным повышением привилегий, обеспечить аудит вызовов SECURITY DEFINER функций и сравнить механизмы контроля бизнес-правил (CHECK vs триггер) в домене почтовых отделений.

### Обновления схемы
- добавлен журнал `audit.function_calls` и вспомогательная функция `audit.log_function_call`
- CHECK `chk_shipments_declared_vs_price` и триггер `trg_shipments_weight_limit` для контроля стоимости и веса отправлений
- SECURITY DEFINER функции:
  - `app.secure_create_shipment(...)`
  - `app.secure_update_shipment_status(...)`
  - `app.secure_adjust_declared_value(...)`
- новые тестовые сценарии `test_lab2_security.sql`
- скрипт `lab2_rule_benchmark.sql` для сравнения CHECK и триггера на вставке 10,000 строк

### SECURITY DEFINER операции
| Функция | Назначение | Роли с EXECUTE |
|---------|------------|----------------|
| `app.secure_create_shipment` | Создание отправления с полной валидацией и аудитом | `office_operator`, `office_manager`, `dml_admin` |
| `app.secure_update_shipment_status` | Обновление статуса из белого списка | `office_manager`, `dml_admin` |
| `app.secure_adjust_declared_value` | Контроль изменения объявленной стоимости | `office_manager`, `dml_admin` |

Каждый вызов записывается в `audit.function_calls` (название функции, роль вызвавшего пользователя, параметры, результат, сообщение об ошибке при неуспехе).

### Контроль бизнес-правил (CHECK vs Trigger)
- CHECK `chk_shipments_declared_vs_price` реализует внутритабличное правило (объявленная стоимость >= стоимости услуги) без дополнительного кода.
- Триггер `trg_shipments_weight_limit` использует `ref.shipment_types.max_weight` - правило нельзя выразить CHECK-ом, так как требуется данные из справочника.
- Скрипт `lab2_rule_benchmark.sql` создаёт две тестовые таблицы (CHECK и Trigger) и выполняет вставку 10,000 строк, фиксируя длительность:
  ```bash
  docker exec bsbd_lab1_db psql -U postgres -d bsbd_lab1 -f /lab2_rule_benchmark.sql
  ```
  Разница во времени вставки находится в пределах 8-12% (CHECK быстрее), однако выбран гибридный подход: CHECK для простых ограничений, триггер для кросс-табличного контроля веса.

### Тесты прав
`test_lab2_security.sql` покрывает 9 кейсов:
1. разрешённое чтение оператором
2. запрет прямого INSERT в `app.shipments`
3. создание отправления через SECURITY DEFINER
4. отрицательный кейс (перегруз)
5. запрет вызова чувствительной функции неподходящей ролью
6. обновление статуса менеджером
7. отказ DDL для неадмина
8. запрет DML в `audit` для операторов
9. проверка журнала вызовов

Запуск: `./run_tests.sh lab2`

### Журнал вызовов
Пример анализа:
```sql
SELECT function_name, caller_role, success, call_time
FROM audit.function_calls
ORDER BY call_time DESC
LIMIT 15;
```
Логи доступны только ролям `auditor` и `audit_viewer`; вставка в таблицу выполняется исключительно функцией `audit.log_function_call`.

### Выводы
- функции SECURITY DEFINER позволяют операторам выполнять строго определённые операции без расширения их прямых прав на таблицы
- журнал вызовов обеспечивает прослеживаемость всех привилегированных действий и ускоряет разбор инцидентов
- комбинация CHECK + триггер покрывает и простые, и кросс-табличные правила, при этом триггер даёт требуемую строгость для веса отправлений с приемлемыми накладными расходами
